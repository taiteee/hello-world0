# ソフトウェア工学 2026

## git/github演習

[softwareEng Lecture Notes](softwareEng2025.md)

## 第1回
ソフトウェア工学は
- 実務から体系化されている
- 知見と実践が乖離している
- チーム全体に知識が共有されている必要がある

ため難しいとされている．しかし，
- システムの保守や改良を続ける必要がある
- 依存先のパッケージによって修正の必要がある

などの理由で不可欠な分野である．

## 第2回
ソフトウェアの定義は
1. 実行されることで，必要な特性／機能／性能を提供する命令語群
2. プログラムが適切に情報を扱うことを可能とするデータ構造
3. プログラムの操作や使用法を記述した情報

である．また，ソフトウェアの特徴として
- 新しい環境や技術のニーズを満たすよう適応しなければならない
- 新しいビジネス要求を実現するために強化されなければならない
- より近代的なシステムやデータベースと相互運用するため拡張されなければならない
- 進化し続ける環境で実行可能となるよう再構築されなければならない

ソフトウェア工学の定義はIEEEによると下記の通り．  
「ソフトウェアの開発運用，メンテナンスに対するシステマティックで規律ある，定量化できるアプローチの適用；ソフトウェアに対するエンジニアリングの適用」

最終目標はソフトウェアシステムの恒常的な変化に対応する方法論への到達である．

プラクティスの本質：
1. 問題を理解する
2. 解決策を計画する
3. 計画を実行に移す
4. 結果が正しいことを理解する

ソフトウェア工学が必要である理由：
1. ソフトウェアが大規模化・複雑化しており，効率よい開発・メンテナンスが求められる
2. IT人材が不足している
3. 不足の事態に対応した事業・サービスの継続が求められる
4. ニーズが不確実になっており，柔軟な対応が求められる

## 第3回
ソフトウェアライフサイクルについて．
- ソフトウェアの誕生
- ソフトウェアの開発・運用
- ソフトウェアの廃止

開発手順は下記の通り．
### 要件定義
実現する機能としない機能を明確化する．
### 設計
要件定義書からコーディング可能な仕様書へ落とし込む．
### 製作
設計書に従ってシステム構築をする．  
自社開発と外部委託を選ぶ．
### テスト・デバッグ
予めテストしバグを早期に発見したい．  
### 運用・保守
稼働後のデータを活用する．

## 第4回
プロジェクトはルーティンワークではなく，かつ独自性があるものである．

成長の方法としてフォアキャスティングとバックキャスティングの2種がある．フォアキャスティングは現在の立ち位置を基準に考え，バックキャスティングは目標を基準に考える．開発にはバックキャスティングが向いている．

### ソフトウェア分析
ソフトウェアを可視化する手法である．
- 物量に関する分析
- 実行性能に関する分析
- 開発工数に関する分析
- 品質に関する分析

などに分けられる．

また，ソフトウェアの評価はコードのステップ数やオブジェクト容量などによって判断される．

バグの無いことはバグ発生率によって検証する．
さらにソフトウェアの評価はFP法や使い勝手によって判断される．

### 開発プロセス
- ウォーターフォール型
- スパイラルモデル
- 反復型開発プロセス
- アジャイルプロセス
- スクラム
などが挙げられる．
